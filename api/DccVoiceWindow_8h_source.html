389 // DCC VOICE NG proto
390 
391 //
392 // tcp control connection
393 // --> HELLO: DccVoice protocol header
394 // <-- HELLO: DccVoice protocol header
395 // --> IACCEPT: Codec-description|CodecId,Codec-description|CodecID,Codec... (in order of preference)
396 // <-- IACCEPT: Codec-description|CodecID,Codec-description|CodecId,Codec... (in order of preference)
397 // --> MYADDRESS
398 // <-- MYADDRESS
399 // --> YOURIDIS: <local id for the remote end> (CID)
400 // <-- YOURIDIS: <local id for the remote end> (CID)
401 
402 // Audio is sent in blocks broken in chunks broken in udp packets
403 // Each block is a set of consecutive audio chunks that theoretically
404 // should be played consecutively.
405 // Each packet in a chunk has an ordinal
406 // Chunks must be relatively short in order
407 // to allow a remote end that looses a packet to
408 // synchronize after a short period of time
409 // The maximum number of packets in a chunk is 65535 (but a chunk should be no more than 24-32 KBytes in size
410 // and in general they should be as small as possible, even one packet per chunk, if the codec allows it)
411 // Each chunk should be encoded independently of the others
412 // We can switch codec at each chunk (but not at each packet)
413 // When some packets are lost we loose the entire chunk
414 // A block is completly synchronized in time (unless we loose some chunks: in that case
415 // we may decide to synchronize with silence or insert a glitch...)
416 // Decoding never depends on the future
417 
418 // Each packet should be decompressable (eventually dependently on the previous in the chunk)
419 // but playable independently
420 
421 // start UDP stream
422 
423 // UDP Packet format:
424 
425 // <magic byte>: byte
426 // <magic byte>: byte
427 // <local id>: word
428 // <payload len>: word
429 // <payload>
430 
431 
432 // Payload format:
433 
434 // <codec id>: word
435 // <ordinal in a chunk>: word (0 = beginning of a chunk)
436 
437 // read raw audio data at sample rate X, sample size Y
438 // multiplex data always at this sample rate and sample size
